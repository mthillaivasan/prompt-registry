What is Prompt Registry

Prompt Registry is a governance platform for managing AI prompts in regulated financial services. It ensures every prompt used in production meets regulatory, security, and quality standards before activation.

MVP Journey 1 — New Prompt

An Author creates a new prompt using the Generator. The system runs a compliance check against 17 scoring dimensions across four frameworks (Regulatory, OWASP LLM Top 10, NIST AI RMF, and ISO 42001). Defects are flagged, the Author corrects them, an Approver reviews and activates the prompt. Every action is audit-logged.

MVP Journey 2 — Import & Upgrade

An Author pastes an existing prompt into the Import view. The system analyses it against all scoring dimensions, produces findings and actionable suggestions. The Author accepts, rejects, or modifies each suggestion individually. Once all suggestions are responded to, the system generates an upgraded version and queues a compliance check. The Approver then reviews and activates.

Logging In

Prompt Registry uses JWT (JSON Web Token) authentication. When you log in, the server issues a signed token that is stored in your browser's local storage.

  • Session expiry: Tokens expire after 60 minutes. After expiry, you will be redirected to the login screen automatically.
  • Security: Do not share your credentials or token with anyone. Each action you take is logged against your user identity.
  • Default admin account: admin@promptregistry.local / ChangeMe123! — change this password immediately in production.

If you see "Session expired", simply log in again. Your work-in-progress is not lost — prompts and proposals are persisted server-side.

Creating a New Prompt

Follow these seven steps to create a compliant prompt:

  1. Open the Generator — click "Generator" in the nav bar or "New Prompt" on the Dashboard.
  2. Fill in metadata — title, prompt type (e.g. Summarisation, Governance), deployment target, input/output types, and risk tier.
  3. Write the prompt text — this is the actual system prompt that will be sent to the AI model. Include all instructions, constraints, and output formatting.
  4. Set review cadence — how often (in days) this prompt should be reassessed. Default is 365 days.
  5. Submit — the system creates the prompt in Draft status with version 1. If auto-compliance is checked, a compliance check runs immediately.
  6. Review compliance results — check the gold standard score, fix any blocking defects by creating new versions with corrected prompt text.
  7. Request activation — once all blocking defects are resolved, activate the version to move the prompt from Draft to Active.

Importing an Existing Prompt

The Import & Upgrade workflow follows the suggestion-first rule: the system never silently rewrites your prompt. Every change is proposed as a suggestion that you must explicitly accept, reject, or modify.

  1. Paste your prompt — open the Import view and paste the existing prompt text into the text area.
  2. Submit for analysis — the system runs an injection scan, then calls the AI to classify the prompt and score it against all 17 dimensions.
  3. Review findings — each finding shows the dimension, current score, severity, and specific gap identified.
  4. Respond to each suggestion — for every suggestion, click Accept (use suggested text as-is), Reject (skip this change), or Modify (edit the suggested text). Each response is audit-logged immediately.
  5. Apply the upgrade — once all suggestions are responded to, click Apply. The system builds the improved prompt from your accepted/modified text and creates a new version. A compliance check is queued automatically.
  6. Or abandon — if you decide not to proceed, click Abandon with a reason. The proposal is closed and the reason is audit-logged.

The Apply button is disabled until every suggestion has a response. Attempting to apply with missing responses returns HTTP 422.

Running a Compliance Check

A compliance check scores a specific prompt version against all 17 active scoring dimensions using the Claude AI model.

How it works

  • The system assembles a dynamic scoring prompt from active ScoringDimension records at runtime — never hardcoded.
  • Call 1 (Scoring): Claude scores the prompt text against each dimension on a 1–5 scale.
  • Call 2 (Anomaly Detection): A second Claude call validates the scoring output for signs of prompt injection compromise (unexpected persona, inflated scores, missing sections).
  • The job runs asynchronously. You receive a job ID immediately and can poll for the result.

Cached results

Results are cached per version. If you request a check on a version that already has a valid cached result, the cached result is returned instantly without calling Claude. Use force_refresh to bypass the cache.

High-risk sign-off

Prompts with risk tier High or Prohibited require additional scrutiny. Any blocking defect (REGULATORY dimension scoring 1 or 2) prevents the prompt from being activated until the defect is resolved in a new version.

Activating a Prompt

Activation marks a specific version as the production version of a prompt. Four prerequisites must be met:

  1. No blocking defects — all REGULATORY dimensions must score 3 or above (above the blocking threshold of 2). Versions with any REGULATORY score of 1 or 2 cannot be activated.
  2. Compliance check completed — the version must have a completed compliance check. You cannot activate an unchecked version.
  3. Approver cannot be the Author — the person activating (approving) the version should not be the same person who created it. This enforces the four-eyes principle required by financial regulators.
  4. Prompt not Retired — retired prompts cannot have versions activated. The prompt must be in Draft, Active, Review Required, or Suspended status.

When a version is activated, any previously active version for that prompt is automatically deactivated. Only one version can be active at a time. If the prompt was in Draft status, activation moves it to Active.

Gold Standard Score

The gold standard score is a weighted composite of all four framework scores, expressed on a 0–100 scale.

Framework weights

FrameworkWeightDimensions
REGULATORY40%6 dimensions (REG_D1–REG_D6)
OWASP LLM Top 1030%5 dimensions (OWASP_LLM01–LLM09)
NIST AI RMF20%4 dimensions (NIST_GOVERN_1–MANAGE_1)
ISO 4200110%2 dimensions (ISO42001_6_1, ISO42001_8_4)

Calculation

Each framework's average dimension score (1–5) is normalised to 0–1, multiplied by its weight, and summed. The result is scaled to 0–100.

How to correct a defect

  1. Open the prompt in the Detail view and review the compliance scores.
  2. Identify the dimension(s) with low scores and read the rationale.
  3. Create a new version with corrected prompt text addressing each gap.
  4. Run a new compliance check on the new version.
  5. Once all blocking defects are resolved, activate the version.

Every defect, correction, and version change is immutably logged in the audit trail.

Regulatory use

The audit log is designed to satisfy FINMA Circular 2023/1 and MAS TRM requirements for AI governance traceability. Every record includes a timestamp (set by the database, not the application), user ID, entity reference, and action detail in JSON format. Auditors have read-only access to the full log.

Injection Detection

The injection scanner protects the system by checking all user-submitted text against a database of known attack patterns before it is processed or sent to Claude.

How it works

  • Patterns are loaded from the injection_patterns table and cached for 60 seconds.
  • Three match types: substring (case-insensitive), regex (pattern matching), and unicode_range (hidden character detection).
  • 23 patterns across 6 categories: Instruction override, Persona hijack, Exfiltration, Delimiter attack, Unicode manipulation, and Structural anomaly.

Result tiers

ResultSeverityAction
CleanNoneText is wrapped in delimiter tags and processed normally.
SuspiciousMedium or HighText is wrapped, flagged, and processed with caution. Logged for review.
CriticalCriticalText is quarantined. Not sent to Claude. Logged as InjectionDetected. Analysis stops.

The highest-severity match determines the overall result. Critical always wins over High, which wins over Medium.

Frequently Asked Questions

1. Can I edit a prompt version after it's created?

No. Prompt versions are immutable by design — enforced by database triggers. To make changes, create a new version. This ensures a complete audit trail of every iteration.

2. What happens if the compliance check fails?

A "Fail" result means one or more REGULATORY dimensions scored 1 or 2 (blocking defects). The prompt cannot be activated until you create a corrected version that scores 3 or above on all regulatory dimensions.

3. How long does a compliance check take?

Typically 10–30 seconds. The check makes two Claude API calls (scoring + anomaly detection). Results are cached per version, so subsequent checks on the same version return instantly.

4. Can I use force_refresh to bypass the cache?

Yes. Set force_refresh: true when submitting a compliance check to ignore the cache and re-score. This is useful after scoring dimensions have been updated.

5. Why can't I activate a version?

Check: (a) no blocking defects remain, (b) a compliance check has completed, (c) the prompt is not Retired, and (d) ideally the approver is a different person from the author.

6. What does "Pass with warnings" mean?

The prompt has no blocking defects (all REGULATORY scores are 3+), but one or more dimensions across any framework scored below 4. The prompt can be activated, but improvement is recommended.

7. How do I add or modify scoring dimensions?

Scoring dimensions are managed in the scoring_dimensions database table. Admin users can update dimension descriptions, criteria, or active status. Changes take effect on the next compliance check. Cached results for existing versions are not automatically invalidated — use force_refresh to re-score.
